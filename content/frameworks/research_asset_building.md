# Research Asset Building

## 定位

本框架用于在 Profile Diagnosis（见 `content/frameworks/profile_diagnosis.md`）识别出缺口后，针对四类资产缺口制定补强方案。执行顺序见 `content/decision_rules/research_asset_priority.md`。

---

## 时间窗口 vs. 可建设资产

| 距申请截止 | 可有效建设的资产类型 |
|---|---|
| 12+ 个月 | 全部（含新推荐关系建立、实质研究参与）|
| 6-12 个月 | 推荐关系深化、研究经历展示、Coding 样本完善、叙事重构 |
| 3-6 个月 | 推荐关系只能强化（不能从零建立）、Coding 快速补充、叙事精炼 |
| 1-3 个月 | Coding 样本优化、叙事微调、现有经历包装 |
| <1 个月 | 只能整合现有材料，不宜尝试建立新信号 |

---

## 四类缺口与补强路径

### 缺口 A：推荐信（Recommendation）

**诊断标志**：无强力推荐人 / 推荐信组合不理想 / 推荐人拒绝推荐

**补强路径：**

**路径 A1：现有关系深化**  
与现有弱/中等推荐人重新发起实质合作（代码协助、文献梳理、数据清理），让对方有具体可叙述内容可写。  
*适用：距截止 3 个月以上，推荐人本人有意愿*

**路径 A2：新推荐关系建立**  
通过 Cold Email、课程助理、研究项目参与建立新推荐关系。  
*适用：距截止 6 个月以上，且推荐人具备学术公信力*  
*红线：时间不足时建立新关系，不如不建立。关系未成熟的推荐信比没有更有害。*

**路径 A3：推荐信组合优化**  
在已有推荐人中调整优先级，选最有分量的组合；与弱推荐人沟通降级为 CC 或 informal reference。  
*适用：所有时间窗口*

**路径 A4：目标市场重定向**  
如果在当前时间窗口内无法完成推荐信补强，转入 Profile Diagnosis → Dimension 4（Positioning）重新评估目标子市场，将资源集中在推荐信要求较低或推荐人网络更匹配的方向。  
*适用：距截止 <3 个月且路径 A1-A3 均不可执行*

---

### 缺口 B：研究经历（Research Experience）

**诊断标志**：无 RA 经历 / 研究经历不可叙述 / 经历与申请方向不相关

**补强路径：**

**路径 B1：现有经历深度挖掘**  
将已有研究参与经历（无论深浅）转化为可叙述的信号：提取做了什么、遇到什么问题、如何解决，构建具体的"研究行动叙事"。  
*适用：所有时间窗口*

**路径 B2：Replication 补充**  
选择目标 PI 方向相关的经典论文做 Replication，作为展示技术能力和研究理解力的替代信号。  
*适用：距截止 2 个月以上，见 `content/frameworks/replication_walkthrough.md`*

**路径 B3：参与在研项目**  
通过现有导师或合作方进入一个活跃研究项目，即使是边缘性参与（数据清理、文献综述）也可产生叙事素材。  
*适用：距截止 4 个月以上*

**路径 B4：接受跳板逻辑**  
在现有研究经历确实薄弱时，诚实定位为"处于研究经历积累初期的候选人"，主攻对研究经历要求相对低的职位，以此为跳板积累。

---

### 缺口 C：Coding 能力（Coding）

**诊断标志**：无 Coding 样本 / 样本质量低 / 不适配岗位技术栈

**补强路径：**

**C1：Replication 样本**  
将 Replication 过程完整代码化，整理为清晰注释的 GitHub repo，同时补强经历缺口（见 B2）。

**C2：Coding Exercise 专项训练**  
针对目标岗位常见 Coding 题型做定向练习，重点是"边做边说"的 walkthrough 能力，而非算法深度。  
见 `content/frameworks/ai_assisted_coding.md`

**C3：AI 辅助工作流展示**  
如果原生 Coding 能力有限，展示"AI 辅助下的高质量研究代码输出"作为能力叙事，见 `content/decision_rules/ai_usage.md`。

---

### 缺口 D：叙事一致性（Narrative）

**诊断标志**：Profile Diagnosis Dimension 3 识别为"伪主线型"或"碎片型"

**补强路径（4 步）：**

**D1：识别底层研究问题**  
从所有经历中提炼"你一直在追问的那个研究问题"——不是主题，而是问题结构。  
工具：见 `content/frameworks/research_interest_narrative.md`

**D2：Reframe as Extension**  
将方向转变、跨领域经历重新定义为"底层问题的延伸探索"，而非"方向改变"。  
工具：见 `content/frameworks/reframe_as_extension.md`

**D3：素材重新排序**  
用底层研究问题作为叙事主线，重新安排经历的呈现顺序（不是时间顺序，是逻辑顺序）。

**D4：一致性检验**  
对所有材料（CV 经历描述、Cover Letter、研究兴趣陈述）执行一致性检验，确保三者指向同一叙事主线。  
原则：见 `content/frameworks/core_principles.md`（主线一致性原则）

---

## 补强优先级

执行顺序见 `content/decision_rules/research_asset_priority.md`。

简要规则：工作授权约束前置 → 推荐信优先 → 研究经历 → 叙事并行修复 → Coding 最后。

---

## 典型失败模式

**FM-1：四个缺口同时启动补强**  
资源分散，每项都浅尝辄止。正确做法：按优先级顺序，集中资源逐个解决。

**FM-2：时间不足时建立新推荐关系**  
关系未成熟时产出的推荐信质量，往往低于不推。

**FM-3：用叙事掩盖真实信号缺失**  
叙事补强的前提是已有基础信号，叙事本身无法创造信号，只能放大或整合信号。

**FM-4：Coding 补强用力过猛**  
Coding 是门槛性信号，不是差异化信号。在推荐信和研究经历未解决时，过度投入 Coding 训练不产生竞争优势。
